# Suno Mall 项目优化计划

## 一、架构优化

### 1.1 拆分单体架构
- **现状**: 当前为 Spring Boot 单体架构，包含 123 个 Java 文件，涉及认证、回收、商城、支付、评价等多个业务模块
- **问题**: 
  - 单应用部署，资源争用严重
  - 服务间耦合度高，无法独立扩展
  - 故障隔离困难，一处故障影响整体
- **建议**:
  - 将认证服务拆分为独立模块（`auth-service`）
  - 将核心业务（回收、商城）拆分
  - 将支付服务独立部署
  - 使用 Feign 进行服务间通信

### 1.2 数据源优化
- **现状**: H2/MySQL 切换仅通过配置文件
- **问题**: 
  - 开发环境 H2 内存库导致测试数据丢失
  - 多数据源切换不优雅
- **建议**:
  - 引入 Flyway/Liquibase 管理数据库迁移
  - 实现数据库连接池自动配置（HikariCP）
  - 添加读写分离支持

## 二、数据库优化

### 2.1 索引优化
- **问题**: 部分查询字段缺少索引
- **建议**:
  - 为 `RecycleOrderEntity` 的 `status`, `created_at` 添加联合索引
  - 为 `ResaleListingEntity` 的 `is_on_sale`, `created_at` 添加索引
  - 为 `ResaleOrderEntity` 的 `status`, `user_id` 添加联合索引
  - 为 `PaymentNonceEntity` 的 `key` 添加唯一索引

### 2.2 查询优化
- **现状**: `DatabaseOptimizationHelper` 已实现分批查询，但部分查询仍有 N+1 问题
- **问题**:
  - 控制器层可能存在 N+1 查询
  - 分页查询可能返回过多数据
- **建议**:
  - 使用 `JOIN FETCH` 优化关联查询
  - 实现二级页面查询（先查分页，再批量查详情）
  - 添加查询结果缓存（Redis）

### 2.3 事务优化
- **现状**: 12 个服务类使用 `@Transactional`
- **问题**:
  - 事务范围过大，影响并发性能
  - 缺少超时配置
- **建议**:
  - 拆分大事务为小事务
  - 添加 `@Transactional(timeout = 30)` 超时配置
  - 使用 `REQUIRED` 和 `REQUIRES_NEW` 明确事务传播行为

## 三、性能优化

### 3.1 JPA 优化
- **问题**: 频繁实体加载导致性能问题
- **建议**:
  - 使用 `@Query` 自定义查询避免 N+1 问题
  - 对查询字段添加 `@ElementCollection` 减少实体大小
  - 实现 ChangeTrackingStrategy（检测变更）

### 3.2 缓存策略
- **现状**: 配置中心支持 ETag 缓存，但未实现
- **建议**:
  - 引入 Redis 缓存热点数据（商品列表、订单状态）
  - 实现读写穿透缓存
  - 添加缓存过期和更新机制

### 3.3 并发优化
- **问题**: 乐观锁重试机制存在竞态条件
- **建议**:
  - 使用 `CAS` 操作减少竞争
  - 实现分布式锁（Redisson）
  - 优化重试策略，避免线程饥饿

## 四、代码质量优化

### 4.1 减少 Magic Numbers
- **问题**: 代码中多处使用魔法数字
- **建议**:
  - 从配置文件中读取业务参数
  - 使用常量定义（如 `Constants.MAX_RETRY_ATTEMPTS`）

### 4.2 异常处理优化
- **现状**: 使用 `IllegalArgumentException`
- **问题**: 违反 Spring 异常处理最佳实践
- **建议**:
  - 使用自定义业务异常（如 `ResourceNotFoundException`）
  - 统一异常处理（`GlobalExceptionHandler`）

### 4.3 REST 响应标准化
- **问题**: 部分接口返回原始数据
- **建议**:
  - 统一使用 `ApiResponse` 包装数据
  - 添加分页信息（总记录数、总页数）

## 五、安全性优化

### 5.1 JWT 优化
- **问题**: 
  - JWT 密钥硬编码在配置文件中
  - Refresh Token 验证不够严格
- **建议**:
  - 将密钥移至环境变量
  - 实现 Refresh Token 吊销机制
  - 添加 JWT 有效期根据角色调整

### 5.2 SQL 注入防护
- **问题**: 部分查询可能存在 SQL 注入风险
- **建议**:
  - 对所有用户输入进行白名单验证
  - 使用参数化查询

## 六、可观测性优化

### 6.1 日志增强
- **现状**: 缺少结构化日志
- **建议**:
  - 引入 Micrometer 进行指标采集
  - 添加请求追踪 ID（Trace ID）
  - 实现日志分级（INFO/WARN/ERROR）

### 6.2 监控告警
- **现状**: 缺少系统监控
- **建议**:
  - 接入 Prometheus + Grafana
  - 监控 JVM 状态、数据库连接池
  - 告警关键指标（错误率、延迟、吞吐量）

## 七、测试优化

### 7.1 单元测试增强
- **现状**: 缺少核心 Service 的单元测试
- **建议**:
  - 为所有 Service 添加单元测试
  - Mock 外部依赖（Provider 层）
  - 覆盖边界条件

### 7.2 集成测试优化
- **问题**: H2 数据导致测试不稳定
- **建议**:
  - 使用独立测试数据库
  - 实现测试数据工厂（Test Data Fixtures）
  - 添加 CI/CD 测试流程

## 八、部署优化

### 8.1 Docker 化
- **建议**:
  - 编写多阶段 Dockerfile
  - 最小化镜像体积（使用 Eclipse Temurin）
  - 添加健康检查

### 8.2 配置管理
- **建议**:
  - 使用 Spring Cloud Config 统一管理配置
  - 添加配置加密
  - 实现配置热更新

## 优化优先级建议

| 优先级 | 优化项 | 理由 |
|--------|--------|------|
| **P0** | SQL 索引优化 | 直接影响查询性能 |
| **P0** | 索引添加与查询优化 | 减少数据库压力 |
| **P1** | 事务拆分 | 提高并发性能 |
| **P1** | 缓存引入 | 减少数据库访问 |
| **P2** | 架构拆分 | 提高可维护性 |
| **P2** | 监控告警 | 便于问题排查 |
| **P3** | 架构拆分 | 中长期规划 |
